# SBIR/STTR Platform Refinement — Implementation Plan

## Vision
Narrow GovWin from a general federal contracting platform to a **focused SBIR/STTR opportunity finder + proposal build service** for small businesses pursuing federal research funding.

## Business Model
- **$199/mo base** — Finder + Minder (opportunity discovery, scoring, alerts, pipeline tracking)
- **$499 per Phase I proposal build** — Expert-reviewed template + AI reanimation workspace
- **$999 per Phase II proposal build** — Same, larger scope
- **Future: White Glove + 5% success fee** — Full BD/capture team (not in V1)
- **72-hour cancellation window** on proposal purchases, 1-week template delivery commitment

## Collaboration & Permissions Architecture

### Section-Level Permissions
Permissions JSONB on proposal_collaborators enhanced to per-section granularity:
```jsonc
{
  "default": "view",
  "sections": {
    "tech_approach": "edit",     // can edit
    "abstract": "view",          // read-only
    "budget": "none",            // hidden
    "past_performance": "review" // view + comment
  },
  "uploads": { "can_upload": true, "can_delete_own": true, "can_view_all": false },
  "library": { "can_access": false },  // partners: no library
  "proposal": { "can_view_metadata": true, "can_advance_stage": false }
}
```
Permission levels: Hidden / View / Review (view+comment) / Edit
Quick-set presets: Writer, Reviewer, Research Partner, Volume Lead

### Liability & Consent Chain
When admin invites partner → consent_record (partner_data_responsibility) from admin
When partner accepts → consent_record (partner_terms_of_service) from partner
When partner first accesses → consent_record (ai_disclosure) from partner
All immutable, timestamped, IP-addressed in existing consent_records table.

### Unified Audit Trail
customer_events enhanced with actor_type (user/partner/admin/ai_agent/system) + actor_label.
ALL actions logged: admin template builds, AI section population, partner uploads, scoring runs.
Correlation chains link related events (template delivery → section population → AI drafting).

## Partner Access Architecture Notes

**Design principle:** Partners are real users with real accounts, but scoped to specific proposals.

**User lifecycle:**
1. Tenant admin invites partner (email + proposal assignment)
2. Partner receives invite link, creates account (role=partner_user)
3. Partner logs in → middleware detects partner_user role → redirects to partner portal view
4. Partner portal: shows ONLY assigned proposals (query filters by proposal_collaborators)
5. Admin revokes access → partner_access_grants.revoked_at set → partner sees nothing
6. If partner is invited to another proposal later, same account gets new grant

**Key architectural decisions:**
- partner_user has tenant_id set to the FIRST tenant that invited them (for DB FK integrity)
- Multi-tenant partner access (same partner on proposals in different tenants) handled by partner_access_grants table — one row per tenant+proposal grant
- verifyProposalAccess() is the ONLY new security function; verifyTenantAccess() unchanged
- Existing proposal_collaborators.permissions JSONB controls what partner can do (no new permission model needed)
- proposal_workspace_files already scoped to proposal_id — no additional file security changes needed
